# 口述转 HTML 演讲 PPT Skill

## 状态

试点 / 已有月亮个人实践基础 / 适合继续用真实演讲验证

## 一句话

把演讲录音、口述思路、AI 追问、飞书文档和图片素材，整理成适合现场表达的 HTML 网页 PPT。

## 目标用户

- 内容创作者
- 社区主理人
- 独立工作者
- 讲师 / 工作坊组织者
- 需要快速准备公开分享、课程、路演或内部汇报的人

## 真实痛点

很多人准备演讲时，最难的不是“没有内容”，而是：

- 脑子里有很多判断，但还没有形成演讲结构。
- 口述很快，整理成 PPT 很慢。
- 文档、图片、案例和讲稿散在不同地方。
- 传统 PPT 模板容易把表达变得僵硬。
- AI 能写大纲，但很容易把演讲稿变成文章。

## 工作流

1. 先用录音或口述把脑子里的逻辑说出来。
2. 让 AI 抽取演讲主线、听众张力、核心判断和结构节奏。
3. 通过几轮追问，把缺失判断交回给演讲者确认。
4. 整理成飞书 / 文档可承载的 slide outline。
5. 补充图片、截图、案例、引文和视觉参考。
6. 选择合适的 HTML PPT 模板或视觉方向。
7. 生成固定 16:9 舞台的 HTML 网页 PPT。
8. 浏览器验证，检查桌面、手机、文字溢出、图片加载和演讲节奏。

## 输入材料

- 演讲录音或转写
- 口述笔记
- 飞书文档或大纲
- 目标听众
- 演讲时长
- 图片、截图、logo、案例或视觉参考
- 希望达到的现场效果

## 输出物

- 演讲主线地图
- slide outline
- speaker notes
- HTML deck build brief
- 可打开的 HTML 网页 PPT
- 可选 PDF 导出或分享链接

## 人机边界

AI 可以做：

- 整理口述逻辑
- 提出追问
- 拆分 slide rhythm
- 生成 slide outline
- 建议视觉结构
- 生成 HTML deck 初稿
- 做事实、视觉和去 AI 味检查

人必须判断：

- 这场演讲真正要让听众记住什么
- 哪些内容适合放在屏幕上
- 哪些案例、图片、人物和合作信息可以公开
- 哪些表达要保留现场感
- 最终演讲节奏是否符合本人表达方式

## 质量门

- 逻辑门：听众、张力、主线和结构节奏清楚。
- 演讲门：屏幕文字服务口头表达，而不是替代演讲。
- 素材门：图片、案例、截图和引用有来源与公开边界。
- 事实门：名称、时间、数据、身份和 logo 准确。
- 视觉门：HTML deck 固定 16:9，桌面和手机打开不变形。
- 去 AI 味门：没有泛泛总结、口号式结尾和过度工整结构。

## 为什么值得产品化

这个工作流比普通 PPT 生成更有复用价值，因为它产品化的是“演讲准备过程”，不是“生成幻灯片”。

它适合创作者和独立工作者：先用口述保护真实思路，再用 AI 帮忙整理、追问和转译，最后通过 HTML deck 获得更灵活的视觉呈现。

## 当前风险

- 需要更多真实演讲案例验证。
- 对图片素材和视觉模板依赖较高。
- 如果没有浏览器验证，容易出现文字溢出或视觉拥挤。
- 需要防止把演讲稿写成文章，再硬拆成 slides。
