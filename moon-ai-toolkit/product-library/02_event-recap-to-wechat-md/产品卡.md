# 活动复盘转公众号 Markdown Skill

## 状态

可对外展示 / GitHub 已发布

## 一句话

把活动复盘、公开报道和现场材料整理成公众号 Markdown 与会后传播包。

## 目标用户

- 活动组织者
- 社区主理人
- 品牌活动负责人
- 伙伴传播负责人

## 真实痛点

活动结束后，材料散落在录音、群聊、照片、伙伴记录和公开报道里。如果不及时整理，现场价值很快沉没。

## 交付承诺

将活动口述复盘、同场公开报道、伙伴记录和图片整合成公众号 Markdown、会后传播包、社区记忆包或伙伴回顾包。

## 月亮的方法

这条工作流来自 2050、AMD、Datawhale 等活动报道和会后复盘经验。

月亮的方法是把活动结束后的零散材料先整理成“现场发生了什么、谁参与了、大家为什么记得、哪些内容可公开、哪些价值值得留下”。如果已有前期报道、伙伴报道、社交平台反馈或用户体验，先做调研地图，交给使用者判断是否写入正文。会后稿的重点是把小范围现场转译成公域读者能理解的社区记忆和关系 proof。

## 工作流

1. 收集口述复盘、照片、群聊反馈、伙伴记录、公开报道和社媒反馈。
2. 做素材地图：时间线、人物、现场画面、反馈、可引用素材和不可公开内容。
3. 判断输出形态：会后报道、现场观察、伙伴传播包、社区记忆包。
4. 起草 Markdown 图文稿，保留图片位置和图注建议。
5. 做事实、图片授权、公开边界和去 AI 味检查。
6. 回收成活动案例、伙伴 proof 和下一次活动的素材库。

## 最小交付物

- 公开资料调研地图
- 战略叙事诊断
- 公众号 Markdown 草稿
- 图片位置与图注建议
- 发布前确认
- 活动资产回收卡

## 人机边界

AI 可以做调研地图、素材整理、结构建议、Markdown 草稿和检查清单。

人必须判断哪些现场细节可公开、哪些伙伴记录能引用、哪些照片能用、活动真正值得被记住的是什么。

## 验证等级

L2 完整交付。已有多个活动报道经验和 GitHub skill。下一步目标是 L3：让活动组织者按 README 和材料清单独立跑出一版会后传播包。

## 质量门

- 现场门：有没有真实的人、画面、反馈和关系。
- 来源门：每条素材来源清楚。
- 授权门：照片、群聊和伙伴记录可公开。
- 记忆门：文章留下活动真正值得被记住的东西。
- 声音门：不把会后稿写成空泛感谢和流程流水账。
