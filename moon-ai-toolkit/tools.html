第一步不是写正文，是把问题摊开

很多稿子卡住，不是因为不会写，而是因为材料、读者、目的和边界还没理清楚。第一轮先让 AI 说明：这篇稿要完成什么任务，哪些材料能用，哪些事实要核对，哪种结构更合适。

  1. 01复制“写稿前整理指令”。
  2. 02粘贴到 Claude Code 或 Codex，也可以放进你熟悉的 AI 对话里。
  3. 03把方括号里的示例换成你的真实材料、链接或文件路径。
  4. 04先看稿件方案，确认方向后再让它写第一版。

你可以带来什么

  1. 01材料：策划案、录音转写、会议纪要、旧稿、网页链接、图片说明。
  2. 02任务：公众号、活动稿、人物稿、演讲 PPT、作品集案例、商业合作稿。
  3. 03边界：哪些必须保留，哪些不能公开，哪些事实要先确认。

第一轮会得到什么

  1. 01这篇稿真正要完成的沟通任务。
  2. 02材料使用清单：能用在哪里、支撑什么、哪些要核对。
  3. 03下一步建议：现在可以写第一版，还是要先补材料、先校对边界。

拿到结果后怎么走

  1. 01材料够：让 AI 按这张方案写第一版。
  2. 02材料不够：先回答它列出的 3-5 个关键问题。
  3. 03准备发布：最后再过事实校对、去 AI 味和公开边界。

一个真实使用场景

比如你手上有一场活动的策划案、会议纪要、嘉宾名单和合作方要求，但还不知道公众号文章应该怎么写。

第一轮先得到一张这样的方案

  1. 01这篇稿的任务：报名召集、合作价值说明、会后记录，还是项目解释。
  2. 02材料怎么用：哪些放开头，哪些支撑主体，哪些只能作为背景。
  3. 03风险和缺口：时间地点、嘉宾身份、合作方口径、未确认信息。
  4. 04下一步：直接写初稿，或先补 3 个问题，再进入成稿。

为什么还要讲人机边界

内容不是越快生成越好。尤其是商业合作稿、活动报道和人物稿，事实、关系分寸和公开边界都需要人来判断。这里的 Human in the loop，不只是“人最后检查一下”，而是从一开始就把人的判断放进流程里。

01 / Before Writing

写稿前：先把材料和边界理清楚

适合手上已经有材料，但不知道怎么开写的人。重点不是马上生成正文，而是先建立一张能判断方向、追问缺口、检查事实的稿件地图。

02 / Community Events

活动传播：把现场和合作变成可审核稿件

适合社区、活动、会议和生态合作场景。共同问题是：信息很多、口径复杂、合作方敏感，写之前必须先弄清楚哪些能写、为什么写、写给谁看。

活动预告 / 分论坛介绍 / 商业合作稿

社区活动内容传播工作台

把策划案、会议纪要、议程和合作方口径，整理成可审核的活动传播稿。

它帮谁、解决了什么痛点？

帮社区运营者、DevRel、活动负责人和临时接手稿件的实习生。它解决的是“材料有了，但不知道活动稿到底要召集、解释价值，还是给合作方交付 proof”的问题。

它是怎么一步步做的？

输入：策划案、议程、嘉宾信息、会议纪要、合作方要求。步骤：先做材料地图，再判断传播任务，检查已有报道和伙伴口径，给结构与缺口。输出：可继续交给编辑审核的活动稿方案或初稿。

它的设计思路里，有哪些值得我学习？

它没有把活动信息按议程罗列，而是先问“这篇稿在帮谁完成什么沟通任务”。值得学的是：商业合作稿不能只堆信息，要同时照顾合作方价值、事实边界和读者是否真的关心。

会后报道 / 活动复盘 / 社区记忆

活动复盘转公众号

把照片、群聊反馈、口述复盘、公开报道和伙伴记录，整理成会后传播包。

它帮谁、解决了什么痛点？

帮活动组织者、社区主理人和需要给伙伴交付回顾材料的人。它解决的是“活动结束后素材散在照片、群聊、口述和公开报道里，很难快速变成一篇可信回顾”的问题。

它是怎么一步步做的？

输入：复盘口述、照片说明、群聊反馈、公开报道、伙伴记录和图片边界。步骤：整理时间线、人物、现场画面和反馈，标注来源可靠度，判断适合报道、观察稿还是传播包。输出：会后稿和素材地图。

它的设计思路里，有哪些值得我学习？

它最值得学的是“来源意识”。群聊感受不能直接写成事实，照片也要有公开边界。好的活动复盘不是把热闹写满，而是把可信素材、现场价值和后续资产回收串起来。

03 / Source To Output

长素材再生产：把录音、口述和思路变成内容

适合播客、直播、访谈、圆桌和演讲准备。关键不是把长材料压缩成摘要，而是先判断它更适合变成文章、对谈稿、短观察、素材卡，还是演讲结构。

播客 / 录音 / 视频回放 / 直播

播客音视频转公众号

把音视频内容转成公众号文章、对谈稿、短观察或可继续复用的素材资产。

它帮谁、解决了什么痛点？

帮播客主、直播组织者、访谈者和内容团队。它解决的是“长音视频有价值，但二次整理太费人，直接丢给 AI 又容易变成空泛总结”的问题。

它是怎么一步步做的？

输入：逐字稿、摘要、关键片段、说话人身份、发布平台和授权边界。步骤：先做话题地图和人物观点地图，再判断适合对谈稿、观察文、知识整理还是素材卡。输出：公众号稿或可复用素材包。

它的设计思路里，有哪些值得我学习？

它不强迫所有素材都变成长文章。值得学的是：先判断内容形态，再进入写作；保留有信息量的原话；把身份、引用和授权边界提前写进流程。

演讲 / 口述思路 / HTML 网页 PPT

口述转 HTML 演讲 PPT

把一段口述思路，整理成能现场讲、能浏览器打开的 HTML 网页 PPT。

它帮谁、解决了什么痛点？

帮经常要演讲、做分享、带工作坊的人。它解决的是“脑子里有逻辑，但做 PPT 要从空白页开始，既慢又容易丢掉表达节奏”的问题。

它是怎么一步步做的？

输入：口述录音或转写、飞书文档、听众画像、演讲时长、图片素材和视觉参考。步骤：先做演讲主线地图，再整理 slide outline，最后进入 HTML deck。输出：可讲述的网页 PPT 和发布前检查清单。

它的设计思路里，有哪些值得我学习？

它把“屏幕上出现什么”和“口头讲什么”分开。值得学的是：从听众张力出发搭结构，不急着做页面；先确定主线、节奏和素材，再生成视觉载体。

04 / Public Proof
