# 工作流雷达规则

## 核心原则

雷达不是自动上架器。

它的工作是发现可能有价值的内容工作流，帮月亮节省“从一堆聊天和项目里捞东西”的时间。进入公开工作台前，必须经过人工确认。

## Multi-agent 分工

### 1. 发现 Agent

负责从材料里找信号。

关注：

- 反复出现的操作步骤。
- 用户明确说“这个可以沉淀”“这个可以产品化”“这个可以给别人用”。
- 已经在真实项目里跑过的流程。
- 能解决一类人的真实问题，而不只是某一次临场发挥。

输出：

- 候选名称。
- 来源位置。
- 为什么可能值得封装。
- 原始片段摘要。

### 2. 成熟度 Agent

判断候选是否足够成形。

评分维度，每项 0-2 分：

| 维度 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 场景清晰 | 不知道给谁用 | 有大概方向 | 有明确使用场景和用户 |
| 输入清晰 | 不知道要给什么材料 | 有部分输入 | 输入材料稳定、可列清单 |
| 输出清晰 | 结果不明确 | 有大概结果 | 产出物可交付、可检查 |
| 复用证据 | 只出现一次 | 有复用可能 | 已经跑过多次或有真实案例 |
| 人工边界 | 全靠 AI 生成 | 有少量提醒 | 明确哪些地方要人判断 |
| 质量门 | 没有校对 | 有泛泛检查 | 有事实校对、边界、去 AI 味等检查 |

建议：

- `0-4`：不进入候选池。
- `5-7`：进入 `hold`，继续观察。
- `8-10`：进入 `candidate`，等待月亮确认。
- `11-12`：可以建议进入 `pilot`，但仍需月亮确认。

### 3. 边界 Agent

判断是否适合公开。

必须拦下：

- 含有未脱敏客户信息、合作方内部信息、私聊内容。
- 依赖月亮个人关系或私人判断，别人无法复用。
- 会让用户误以为 AI 可以替代事实确认、采访授权、商业审核。
- 只有漂亮概念，没有真实步骤。

输出：

- 可公开部分。
- 必须删掉或改写的部分。
- 需要月亮确认的问题。

### 4. 封装 Agent

把通过初筛的候选整理成工作流卡。

必须包含：

- 工作流名称。
- 适用场景。
- 目标用户。
- 输入材料。
- 操作步骤。
- 人机边界。
- 质量检查。
- 启动说明。
- 建议状态。

### 5. 上架 Agent

只在月亮确认后工作。

可执行动作：

- 加入网页工作流清单。
- 创建 GitHub Skill 包。
- 写入产品库。
- 标记为 `pilot` 或 `live`。

不能执行：

- 未经确认直接公开。
- 把原始对话、客户材料或未脱敏案例放到公开页。
- 把 `candidate` 误写成已经验证的产品。

## 月亮确认门

任何候选进入公开页之前，必须回答：

1. 这个工作流解决的是一类人的问题，还是一次具体项目的临时问题？
2. 陌生用户拿到它，是否知道要准备什么材料？
3. 输出物是否可检查、可审核、可交付？
4. 它是否保留了人的判断，而不是只给 AI 一个长 prompt？
5. 有没有真实项目或至少一次完整验证？
6. 是否需要脱敏、删去合作方信息或弱化案例细节？
7. 月亮是否愿意把它作为自己的方法对外展示？

只有第 1、2、3、4 条都能回答清楚，才可以进入 `pilot`。
