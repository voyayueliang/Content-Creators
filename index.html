月亮的 AI 工具箱 GitHub 总仓库

给内容创作者、社区和小团队的可复用工作流

把真实材料放进去，跑出能交付、能校对、能复用的内容稿。

这里不是提示词合集，也不是作品陈列页。它只回答一件事：你现在遇到什么内容任务，应该用哪个工作流，准备什么材料，怎样部署并开始用。

怎么用

  1. 先从下面选一个最接近的场景。
  2. 按卡片准备材料：策划案、录音、逐字稿、公开资料或草稿。
  3. 打开 GitHub 部署包，装进 Codex，或直接复制启动句试跑。
  4. 交付前过事实校对和去 AI 味两扇门。

选择你的场景

每个工作流都来自真实内容交付。先解决一个具体任务，再慢慢沉淀成自己的内容系统。

活动预告 / 分论坛介绍 / 商业合作稿

社区活动内容传播工作台

适合社区、活动主办方和生态伙伴，把策划案、会议纪要、议程和合作方口径整理成可审核的公众号稿。

打开 GitHub 部署包

解决的痛点

活动信息很多，但传播稿容易写成资料堆砌；商业合作又不能出错，需要把事实、口径和战略价值分清楚。

如何使用

  1. 准备活动基本信息、策划案和嘉宾议程。
  2. 补充合作方希望传达的价值和不能公开的信息。
  3. 先让工作流做事实缺口检查，再进入草稿。
打开社区活动内容传播工作台。

活动类型：[会前预告 / 分论坛介绍 / 商业合作稿]
材料在这里：[粘贴策划案、会议纪要、嘉宾信息或文件路径]
发布平台：[公众号 / 官网 / 社群]

请先做需求收口和事实缺口检查，再决定文章结构。
会后报道 / 活动复盘 / 社区记忆

活动复盘转公众号

适合活动结束后，把现场照片、群聊反馈、口述复盘、公开报道和伙伴记录，整理成会后传播包。

打开 GitHub 部署包

解决的痛点

活动结束后材料很散，热度很快过去；如果不及时整理，现场价值、伙伴 proof 和社区记忆都会流失。

如何使用

  1. 准备复盘口述、照片说明和公开报道链接。
  2. 标注哪些图片、名字和评价可以公开。
  3. 先生成素材地图，再判断写报道、观察稿还是传播包。
打开活动复盘转公众号工作流。

活动已经结束，复盘材料在这里：[粘贴或给路径]
同场公开报道 / 伙伴记录：[链接或说明]
图片边界：[可公开 / 需确认 / 不可公开]

请先整理素材地图，再判断适合写成报道、观察稿还是传播包。
播客 / 录音 / 视频回放 / 直播

播客音视频转公众号

适合播客主理人、直播栏目、活动主办方和长视频创作者，把音视频内容转成公众号文章、对谈稿或素材资产。

打开 GitHub 部署包

解决的痛点

音视频信息密度高，但很难直接变成可读文章；如果只做总结，又会丢掉说话人的语气、关系和现场感。

如何使用

  1. 准备逐字稿、摘要或关键片段。
  2. 说明说话人身份、发布平台和授权边界。
  3. 先判断内容形态，再决定写长文、对谈稿还是短观察。
打开播客音视频转公众号文章工作流。

素材类型：[播客 / 录音 / 视频回放 / 直播 / 访谈 / 圆桌]
素材、逐字稿或摘要在这里：[粘贴内容、链接或文件路径]
我希望输出：[公众号文章 / 对谈稿 / 短观察 / 素材卡 / 不确定]
公开边界：[可公开 / 需匿名 / 需确认 / 不外发]

请先判断内容形态，不要直接写成长文。
公众人物 / 创始人 / Builder Profile

公众人物深度报道工作台

适合基于公开访谈、播客、演讲、文章和社媒资料，写出有事实约束、有阅读透镜的人物深度报道。

打开 GitHub 部署包

解决的痛点

公开资料很多，但容易写成百科、PR 稿或主观判断；这个工作流先做资料查看台，把事实、引用、推断和边界分开。

如何使用

  1. 给出人物姓名和公开材料链接。
  2. 说明发布渠道、读者和不能推断的边界。
  3. 先做资料查看台和文章角度选择，再写逐节草稿。
打开公众人物深度报道工作台。

对象：[人物姓名]
模式：[第三方无采访 / 当事人自述 / 三方协作]
已有素材：[链接 / 文件 / 暂无，请帮我搜集]
发布渠道：[公众号 / Substack / 作品集 / 机构投稿 / 不确定]

请先做人物资料查看台和素材解剖，不要直接写完整长文。

所有工作流最后都要过两扇门

第一扇是事实校对门：名称、日期、身份、引用、授权、商业口径都要对得上材料。第二扇是去 AI 味门：删掉空泛判断、硬凑转折、过度顺滑和没有来源支撑的漂亮话。
