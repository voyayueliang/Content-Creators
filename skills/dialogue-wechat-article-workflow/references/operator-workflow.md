# 通用对谈转公众号工作流

这套工作流适用于不同账号、不同主持人和不同嘉宾组合：机构账号、个人账号、项目账号、社群账号、嘉宾账号的直播、访谈、双人对谈、多人圆桌、播客回放和活动复盘。

## 1. 素材进入

尽量收齐：

- 逐字稿 / 回放链接 / 音频转写
- 海报、主视觉、直播截图
- 主持人、嘉宾、组织方介绍
- 当期发布账号和希望采用的语气
- 是否需要配图、展示作品、导流报名
- 不能公开的内容：学生、客户、商业信息、家庭故事、合作关系

如果素材不完整，先做 `内容判断卡`，不要急着写全文。

如果素材很长、方向还不清楚，先做 `内容查看台`：判断它适合变成对谈、观点文、视频脚本、短帖、素材卡，还是暂时不外发。不要默认把所有材料都展开成长文。

## 2. 建立当期人物表

每期都重新建立，不复用旧称呼。

| 字段 | 说明 |
| --- | --- |
| 公开姓名 | 文章中出现的名字 |
| 对话标签 | 正文中使用的标签 |
| 当期角色 | 主持人、嘉宾、联合主持、运营伙伴、创始人、学员等 |
| 简介来源 | 海报、官网、用户补充、直播自述、待确认 |
| 公开边界 | 是否可露出全名、机构、项目、产品、头像 |

原则：

- 用户补充优先于 AI 推测。
- 海报和官方物料优先于直播里临时自述。
- 不确定头衔就标 `待确认`。
- 中英文名按当期公开称呼使用，不自动替换。

## 3. 判断发布账号语气

先问：这篇是谁发？

| 发布主体 | 导语语气 |
| --- | --- |
| 机构账号 | 用机构观察者 / 出品方语气，不写个人日记 |
| 个人账号 | 可保留第一人称，但要符合本人表达习惯 |
| 嘉宾账号 | 更突出嘉宾观点和经历 |
| 共创账号 | 用中性编辑部语气 |
| 项目 / 社群账号 | 连接活动现场和社群读者 |

同一份对谈，在不同账号上不能只换署名，导语、标题、结尾和 CTA 都要重调。

## 4. 判断输出形态

| 形态 | 适合情况 | 输出 |
| --- | --- | --- |
| 编辑版对谈稿 | 真实来回最有价值 | 4-5 节对谈 |
| 对谈摘要 | 逐字稿太长，需要压缩 | 主题式摘编 |
| 观点观察文 | 有强判断，嘉宾声音不是重点 | 文章化表达 |
| 人物故事文 | 一个嘉宾的转变或经历是中心 | 故事线 |
| 方法提炼文 | 对谈里有可复用方法 | 框架 / checklist |
| 视频号脚本 | 一个场景或问题很抓人 | 口播稿 |
| 素材卡 | 公开边界不清或暂不成文 | 内部资产 |

不要把所有对谈都写成深度文章。对谈本身就是一种成熟输出。

判断时优先问：这份素材最有价值的是“一个观点”，还是“这场对话如何发生”？如果后者更重要，就保留对谈。

## 5. 找主线

用一句话写清楚：读者为什么要读这篇？

好主线通常回答：

- 读者正在困惑什么？
- 这场对谈提供了什么新的看法或现场？
- 哪些案例让抽象问题落地？
- 读完以后，读者会带走什么问题或方法？

推荐逻辑弧：

1. 从真实场景或读者问题进入。
2. 澄清这场对谈最核心的张力。
3. 展开 2-3 个关键判断。
4. 放入案例、原话和现场。
5. 回到读者能使用的问题或行动。

## 6. 标题工作法

不要只给一组标题。至少分三组：

### 读者痛点型

- 当 AI 什么都能做，孩子 / 创作者 / 创业者还需要什么？
- 为什么我们越会用工具，越要先说清楚自己想要什么？

### 场景故事型

- 一场跨时区连麦之后，我们重新理解了……
- 那些喊着「加班」冲回教室的孩子，想完成的到底是什么？

### 方法判断型

- 不是先学工具，而是先找到问题
- 让 AI 成为辅助器，而不是替代你的想法

标题要按当期读者改写。家长、创业者、教育同行、创作者、投资人看到的是不同问题。

## 7. 正文结构

标准公众号对谈稿：

1. 标题候选
2. 摘要
3. 主视觉 / 海报
4. 对谈嘉宾
5. 写在前面
6. 编辑说明
7. 4-5 节对谈正文
8. 配图占位
9. 编辑备注 / 发布前确认
10. 可回收内容资产

每节内部：

```markdown
## 1. 小标题

**主持人：**  
问题或过渡。

**嘉宾：**  
回答，保留原意和说话质地。

**补充发言人：**  
现场观察或补充。
```

## 8. 写在前面

导语要从真实场景进入：

- 直播开始前的状态
- 跨时区连麦
- 活动现场的空间感
- 嘉宾当时的处境
- 一个能把读者带进去的小细节

然后转入本期核心问题。

避免：

- `在 AI 飞速发展的今天……`
- `本期我们邀请到了重量级嘉宾……`
- `这是一场干货满满的对谈……`

更稳：

- 先写场景，再写问题。
- 先写具体的人和话，再写抽象判断。
- 先让读者进入现场，再解释这场对谈为何重要。

## 9. 配图占位

统一格式：

```markdown
【配图占位 01｜图片类型｜用途｜授权提醒：具体说明】
```

每个占位要告诉编辑：

- 找什么图
- 放在这里解决什么阅读问题
- 需要确认什么授权或隐私

## 10. 人机协作节点

不要让 AI 一次性完成所有判断。建议这样分工：

| 节点 | 人负责 | AI 负责 |
| --- | --- | --- |
| 素材进入 | 提供素材、账号、边界 | 整理来源和缺口 |
| 方向判断 | 确认想发什么 | 给出输出形态建议 |
| 结构选择 | 选择主线和标题方向 | 生成结构与标题组 |
| 初稿 | 判断像不像账号 / 人 | 写出可编辑正文 |
| 反馈 | 说哪里不对 | 翻译成修改动作 |
| 终稿 | 确认授权和发布 | 做事实、逻辑、字数检查 |
| 回收 | 选择后续用途 | 生成素材卡和下一轮问题 |

边界原则：

- AI 可以给选项、结构、草稿和检查，不替人确认发布风险。
- 人必须确认账号语气、署名、关系边界、事实可信度、图片授权和最终是否发布。
- 人反馈“不像”“不对”“敏感”“不能这么说”时，AI 要回到边界校准，而不是只润色。
- 边界不清的信息统一标 `待确认`，不要写成确定事实。

## 11. 发布前确认

必须检查：

- 当期人物表是否正确
- 发布账号语气是否正确
- 事实是否可追溯
- 小节之间是否有递进
- 字数是否按可读正文统计
- 标题是否面向真实读者
- 是否过度总结、过度口号化
- 图片是否有授权
- 是否有未成年人、客户、家庭、商业边界
- 是否需要在文末导流
- 是否通过逻辑、证据、声音、关系四道门
- 是否保留了必要的不确定，而不是被写成过度完整的结论

## 12. 内容资产回收

主稿完成后，回收：

| 卡片 | 内容 | 用途 |
| --- | --- | --- |
| 判断卡 | 核心观点 | 标题 / 口播 / 课程理念 |
| 场景卡 | 现场细节 | 开头 / 朋友圈 |
| 原话卡 | 有力量的表达 | 金句 / 海报 |
| 案例卡 | 具体人物或作品 | 小红书 / 招生 / 作品集 |
| 方法卡 | 可复用步骤 | 教研 / 咨询 / 工作坊 |
| 边界卡 | 授权和风险 | 发布检查 |
| 反馈卡 | 评论和私信 | 下次选题 |
