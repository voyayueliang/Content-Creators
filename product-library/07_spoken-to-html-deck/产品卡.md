# 口述转 HTML 演讲 PPT Skill

## 状态

试点 / 已有月亮个人实践基础 / 需要用完整真实演讲继续验证

## 一句话

把一段口述思路，经过 AI 追问、飞书整理、图片入场和 HTML 模板生成，变成可以现场讲、可以浏览器打开的网页 PPT。

## 目标用户

- 内容创作者
- 社区主理人
- 独立工作者
- 讲师 / 工作坊组织者
- 需要快速准备公开分享、课程、路演或内部汇报的人

## 真实痛点

很多人准备演讲时，常常卡在这些地方：

- 脑子里有很多判断，但还没有形成演讲结构。
- 口述很快，整理成 PPT 很慢。
- 文档、图片、案例和讲稿散在不同地方。
- 传统 PPT 模板容易把表达变得僵硬。
- AI 能写大纲，但很容易把演讲稿变成文章。

## 月亮自己的方法

这条工作流来自月亮自己的高效演讲准备方式：

1. 先开一段录音，把脑子里想表达的逻辑完整说出来。
2. 和 AI 互动几轮，让 AI 追问主线、听众张力、例子、删减点和公开边界。
3. 把互动后的结构放入飞书文档，变成可编辑、可补材料的中转台。
4. 加入图片、截图、案例和视觉参考，让内容不只停留在文字上。
5. 选择合适的 HTML PPT 模板，生成网页演讲稿。
6. 在浏览器里排练，检查节奏、文字溢出、图片加载和现场表达。

真正值得产品化的是这套把现场表达、内容判断和视觉呈现接起来的方法。PPT 只是最后的承载物。

## 工作流

1. 判断入口：口述理清、飞书整理、网页生成，还是演讲修订。
2. 如果只有录音或口述，先生成演讲主线地图，不直接生成 PPT。
3. 让 AI 抽取核心问题、听众张力、主要判断和 3-5 个结构节奏。
4. 通过几轮追问，把缺失判断交回给演讲者确认。
5. 整理成飞书 / 文档可承载的 slide outline。
6. 标出屏幕内容、口头讲述、视觉素材和待确认信息。
7. 补充图片、截图、案例、引文和视觉参考。
8. 选择合适的 HTML PPT 模板或视觉方向。
9. 生成固定 16:9 舞台的 HTML 网页 PPT。
10. 浏览器排练与验证，检查桌面、手机、文字溢出、图片加载和演讲节奏。

## 输入材料

- 演讲录音或转写
- 口述笔记
- 飞书文档或大纲
- 目标听众
- 演讲时长
- 图片、截图、logo、案例或视觉参考
- 希望达到的现场效果

## 输出物

- 演讲主线地图
- slide outline
- speaker notes
- HTML deck build brief
- 可打开的 HTML 网页 PPT
- 可选 PDF 导出或分享链接

## 人机边界

AI 可以做：

- 整理口述逻辑
- 提出追问
- 拆分 slide rhythm
- 生成 slide outline
- 建议视觉结构
- 生成 HTML deck 初稿
- 做事实、视觉和去 AI 味检查

人必须判断：

- 这场演讲真正要让听众记住什么
- 哪些内容适合放在屏幕上
- 哪些案例、图片、人物和合作信息可以公开
- 哪些表达要保留现场感
- 最终演讲节奏是否符合本人表达方式

## 质量门

- 逻辑门：听众、张力、主线和结构节奏清楚。
- 演讲门：屏幕文字要服务口头表达，不能替代演讲。
- 素材门：图片、案例、截图和引用有来源与公开边界。
- 事实门：名称、时间、数据、身份和 logo 准确。
- 视觉门：HTML deck 固定 16:9，桌面和手机打开不变形。
- 去 AI 味门：没有泛泛总结、口号式结尾和过度工整结构。

## 为什么值得产品化

这个工作流比普通 PPT 生成更有复用价值，因为它产品化的是“演讲准备过程”。生成幻灯片只是其中一个环节。

它适合创作者和独立工作者：先用口述保护真实思路，再用 AI 帮忙整理、追问和转译，最后通过 HTML deck 获得更灵活的视觉呈现。

它也适合延展成工作坊或陪跑服务：用户把录音、飞书文档和图片交进来，系统先产出演讲主线和 slide outline，再进入 HTML deck 生成与排练修订。

## 当前风险

- 需要更多真实演讲案例验证。
- 对图片素材和视觉模板依赖较高。
- 如果没有浏览器验证，容易出现文字溢出或视觉拥挤。
- 需要防止把演讲稿写成文章，再硬拆成 slides。
- 需要避免把 Moon 的个人判断误包装成通用规则。
